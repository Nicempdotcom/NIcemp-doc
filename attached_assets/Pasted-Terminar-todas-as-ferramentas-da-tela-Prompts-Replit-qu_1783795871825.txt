## 1. Deixar `PromptGenerator.ts` reutilizável para diferentes tipos de tarefa

Arquivo: `artifacts/nicemp-docs/src/services/prompts/PromptGenerator.ts`

Hoje `buildReplitPrompt(input)` sempre monta o mesmo "Objetivo" genérico ("Realizar uma alteração em..."). Adicionar um segundo parâmetro opcional pra permitir customizar o objetivo e o checklist conforme o tipo de tarefa (refatoração, layout, bug, documentação, spec), mantendo TUDO o resto (arquivos permitidos/proibidos, dependências, risco) exatamente como já funciona hoje, porque essa parte já é 100% baseada em dado real:

```ts
export interface PromptTaskOptions {
  /** Rótulo da tarefa, usado no título do prompt (ex.: "Refatoração", "Correção de Bug"). */
  taskLabel?: string;
  /** Substitui o texto padrão da seção "Objetivo da alteração". */
  objectiveOverride?: string;
  /** Itens extras adicionados ao final do checklist de testes. */
  extraChecklist?: string[];
}

export function buildReplitPrompt(input: PromptEntityInput, options: PromptTaskOptions = {}): string {
  const label = KIND_LABELS[input.kind];
  const title = options.taskLabel
    ? `# Prompt para Replit — ${options.taskLabel}: ${input.name}`
    : `# Prompt para Replit — ${label}: ${input.name}`;

  const checklist = options.extraChecklist?.length
    ? [buildTestChecklist(input), ...options.extraChecklist.map((item) => `- [ ] ${item}`)].join('\n')
    : buildTestChecklist(input);

  return [
    title,
    '',
    '## 1. Objetivo da alteração',
    options.objectiveOverride ?? buildObjective(input),
    '',
    '## 2. Arquivos permitidos',
    buildAllowedFiles(input),
    '',
    '## 3. Arquivos proibidos',
    buildForbiddenFiles(input),
    '',
    '## 4. Dependências',
    buildDependencies(input),
    '',
    '## 5. Nível de risco',
    buildRisk(input),
    '',
    '## 6. Checklist de testes',
    checklist,
    '',
    '## 7. Ao concluir',
    '⚠️ Obrigatório: ao final da tarefa, liste TODOS os arquivos que foram criados, modificados ou removidos — mesmo que a alteração pareça pequena. Não omita nenhum arquivo da lista.',
  ].join('\n');
}
```

Confirmar que `GeneratePromptButton.tsx` e qualquer outro lugar que já chama `buildReplitPrompt(entity)` sem o segundo argumento continuam funcionando normalmente (parâmetro é opcional, com valor padrão `{}`).

## 3. Configuração de cada propósito

Cada card usa `EntityPickerPromptDialog` com uma configuração diferente:

- **Refatoração** (`purpose: 'refactor'`): campo extra "O que está difícil de manter hoje / motivo da refatoração" (Textarea). Objetivo gerado: `Refatorar o [tipo] "[nome]" em [local], SEM mudar o comportamento observável para quem usa. Motivo: [texto do campo].` Checklist extra: `"O comportamento antes e depois da refatoração é idêntico para quem usa (validado manualmente)."`

- **Atualização de Layout** (`purpose: 'layout'`): campo extra "O que precisa mudar visualmente" (Textarea). Objetivo gerado: `Atualizar o layout/estilo do [tipo] "[nome]" em [local]. Mudança desejada: [texto do campo]. Não alterar lógica, dados ou comportamento — só a aparência.` Checklist extra: `"Nenhuma lógica ou dado foi alterado, só estilo/layout."`, `"Testado em desktop e mobile (responsivo)."`

- **Correção de Bugs** (`purpose: 'bugfix'`): dois campos extras — "O que está acontecendo (comportamento atual/errado)" e "O que deveria acontecer (comportamento esperado)" (ambos Textarea). Objetivo gerado: `Corrigir um bug no [tipo] "[nome]" em [local].\n\nComportamento atual (errado): [campo 1]\nComportamento esperado: [campo 2]\n\nInvestigue a causa raiz antes de aplicar a correção — não aplique um "band-aid" que só esconde o sintoma.` Checklist extra: `"A causa raiz foi identificada e documentada (não só o sintoma)."`, `"O bug relatado não reproduz mais após a correção."`

- **Documentar Código** (`purpose: 'docs'`, na seção "Prompts de Documentação"): sem campos extras (ou opcionalmente um Select "Nível de detalhe: Básico | Completo"). Objetivo gerado: `Adicionar/melhorar a documentação (comentários, JSDoc, README de pasta se fizer sentido) do [tipo] "[nome]" em [local], explicando o que ele faz, por que existe e como as dependências listadas abaixo se encaixam. Não alterar nenhum comportamento de código, só documentação.` Checklist extra: `"Nenhuma linha de código funcional foi alterada, só comentários/documentação."`

- **Gerar Specs** (`purpose: 'spec'`, na seção "Prompts de Documentação"): campo extra Select "Público-alvo da especificação: Time técnico | Stakeholder não-técnico". Objetivo gerado: `Escrever uma especificação técnica (arquivo .md separado, não altera o código) descrevendo o [tipo] "[nome]" em [local]: o que faz, como se encaixa no módulo "[módulo]", e suas dependências. Público-alvo: [campo selecionado] — ajuste o nível de detalhe técnico conforme isso.` Checklist extra: `"O arquivo de spec foi criado na pasta de documentação do projeto (perguntar ao usuário onde, se não houver convenção clara), sem alterar nenhum arquivo de código."`
